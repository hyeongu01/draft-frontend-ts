§ 01

서비스 개요

한 줄 정의

직장인이 시기별로 이력서를 쌓아가며 공유하고, 또래에게 피드백을 받는 커리어 커뮤니티 서비스.

이력서는 한 번 쓰고 끝나는 문서가 아니라, 커리어가 진행되면서 계속 갱신되는 살아있는 기록이다.

해결하려는 문제

  • 이직 시점마다 이력서를 처음부터 다시 쓰는 마찰
  • 본인 이력서가 객관적으로 잘 쓴 것인지 판단할 채널 부재
  • 같은 직무·연차 동료의 실제 이력서를 참고할 수 없음
  • 1:1 멘토링 서비스는 비용·접근성 문제로 부담

해결 방식

  • 회사·시기별로 이력서를 독립된 카드로 누적 (커리어 버전 관리)
  • 각 이력서를 공개·비공개 단위로 토글, 또래 풀에 노출
  • 가벼운 반응(좋아요)과 저장(보관)으로 인사이트 축적
  • 깊은 피드백은 1:1 채팅, 채팅 컨텍스트에 어떤 이력서에 대한 대화인지 명시

경쟁 서비스 대비 차별화

서비스중심 가치본 서비스와의 차이
잡코리아 · 사람인채용공고이력서 자체를 공유하지 않음
링커리어 · 잇다현직자 멘토링유료, 또래 피드백 아님
LinkedIn네트워킹이력서 버전 관리 부재, 비공개 피드백 어려움

§ 02

타겟 사용자

1차 페르소나

이직을 고려 중이거나, 이미 여러 회사를 거친 2~10년차 직장인. IT·디자인·기획 직군이 초기 집중 대상.

사용 시나리오

  1. 회사를 옮길 때 현 직장에서의 경험을 정리해 새 이력서를 추가한다.
  2. 비슷한 직무·연차의 또래 이력서를 피드에서 둘러본다.
  3. 마음에 드는 이력서에 좋아요, 참고할 것은 보관한다.
  4. 구체적 피드백이 필요하면 작성자에게 채팅을 건다.
  5. 본인 이력서에 다른 사람이 좋아요·보관을 누르면 받은 반응을 마이페이지에서 확인한다.
§ 03

MVP 범위

포함 기능

  1. 회원가입·로그인 (이메일 + 카카오·구글 소셜)
  2. 이력서 작성 (Tiptap 기반 자체 에디터)
  3. 여러 이력서 누적·관리 (시기별 카드)
  4. 피드 (최신순·인기순)
  5. 좋아요
  6. 보관 (북마크)
  7. 1:1 채팅 (실시간)
  8. 마이페이지 (내 이력서 / 보관함 / 좋아요 / 받은 피드백)
  9. 직무 + 연차 필터

제외 기능 (v2 이후)

  • PDF 업로드 — 검색·표준화 어려움, 자체 에디터로 통일
  • 공개 댓글 — 채팅으로 피드백 동선 단일화
  • 대학생 등 신원 인증
  • AI 피드백 · 자동 추천
  • 합격자 이력서 인증 뱃지
PDF 업로드 제외 이유 — 자체 에디터로 통일하면 데이터가 구조화되어 후일 AI 피드백·추천·검색 고도화의 기반이 된다. 초기엔 사용자 마찰이 있지만 장기 가치가 훨씬 크다.
§ 04

사용자 흐름

신규 사용자 — 첫 이력서 작성부터 피드백 수신까지

First-time user flow/login → Google 또는 GitHub 선택 ↓ /auth/callback (OAuth 토큰 교환) ↓ /onboarding → 닉네임 설정 ↓ /me (빈 상태 · 첫 이력서 CTA) ↓ /me/resumes/new → 제목 입력 ↓ /me/resumes/[id]/edit → 섹션 추가 · 공개 토글 ↓ /me (이력서 카드 1개) ↓ / 피드에서 다른 이력서 둘러보기 ↓ /resumes/[id] → "피드백 요청" ↓ /chats/[id] → 첫 대화 시작

기존 사용자 — 새 회사 합류 후 이력서 추가

Returning user flow/me (기존 이력서 카드 N개) ↓ [+ 새 이력서] → /me/resumes/new ↓ 편집 → 새 챕터 카드가 /me에 추가 ↓ 공개 토글 → 피드에 노출

설계 의사결정

  • 이력서 본문은 JSONB — 사람마다 섹션 구조가 다르고 자주 바뀜. 정형 컬럼으로 박으면 마이그레이션 비용이 큼.
  • 검색 필요 필드만 정형 컬럼 — job_role, experience_years, period_start/end만 별도 컬럼.
  • 카운터는 트리거로 자동 업데이트 — like_count, save_count 등을 매 SELECT마다 집계하면 비용↑.
  • 채팅방은 두 사용자 간 1개 — (user_a_id, user_b_id)에 유니크 제약. 정렬 규칙(작은 id가 user_a)으로 중복 생성 방지.
  • 채팅방에 about_resume_id — 피드백이 시작된 이력서를 명시. 채팅 컨텍스트 바에 표시.

인증Supabase Auth이메일·소셜 통합, RLS와 자연스럽게 결합
실시간Supabase RealtimePostgres Changes·Broadcast·Presence

왜 별도 백엔드를 두지 않는가 — Supabase가 Auth, DB, Realtime, Storage, REST API 자동 생성을 모두 제공한다. 1인 MVP에서 별도 백엔드 서버를 운영하면 운영 단위만 2개로 늘고 얻는 게 거의 없다. 무거운 비즈니스 로직이 명확해지면 그때 Edge Functions로 점진적으로 분리.
§ 08

학습 로드맵

Tier 1 — 셋업 단계 (1주차)

  • Next.js App Router — 파일 라우팅, Server vs Client Component, Server Actions, Metadata API
  • Tailwind CSS — cheatsheet 검색하며 만들기 (외울 필요 X)
  • shadcn/ui — 컴포넌트 설치 방식, 본인 코드베이스로 복사되는 구조

Tier 2 — DB와 인증 (2~3주차)

  • PostgreSQL 기초 — SQL 기본, JOIN, JSONB 연산자, 인덱스, 트리거, FTS
  • Supabase Auth — 세션 쿠키, @supabase/ssr 사용법
  • RLS (Row Level Security) — DB 레벨 권한 검증, 정책 패턴
  • Supabase Client SDK — 쿼리 API, 타입 자동 생성

Tier 3 — 기능 단위 학습 (만들면서)

  • Supabase Realtime — 채팅 만들 시점
  • Tiptap — 에디터 만들 시점, StarterKit + 커스텀 노드
  • React Hook Form + Zod — 폼 검증
  • TanStack Query — 필요할 때만 (옵셔널)

Tier 4 — 배포·운영 (런칭 직전)

  • Vercel — GitHub 연결, 환경변수, Preview 배포
  • Supabase 운영 — 마이그레이션, 환경 분리
  • Sentry, PostHog — 5분 셋업
  • Edge Functions — 필요 시점에 학습
학습 안 해도 되는 것들 — Prisma/Drizzle 같은 ORM(supabase-js로 충분), Redux 같은 무거운 상태 관리, tRPC(Server Actions이 비슷한 역할), 초기 테스트 프레임워크 도입.

§ 10

결정 포인트 / 리스크

10.1 개인정보 보호 (최우선)

이력서는 이름·연락처·학력·경력 등 신원 식별이 가능한 민감정보다. 다음 정책을 처음부터 명확히 잡고 시작한다.

  • 공개 시 이름은 닉네임으로만 표시
  • 전화번호·이메일은 절대 노출하지 않음
  • 식별정보 교환은 본인 의지로 채팅을 통해서만 가능
  • 이력서 단위의 공개·비공개 토글 제공

10.2 탐색 가능성

"어떤 이력서를 볼까"에서 막히면 서비스가 작동하지 않는다. MVP에서는 직무 + 연차 두 필터에 집중하고, 학교·회사·기술스택 필터는 v2로 미룬다.

10.3 채팅 스팸·사칭

헤드헌터·기업 사칭 시도가 분명히 발생한다.

  • v1: 신고 + 차단 기능
  • v2: 직장 인증 (회사 이메일 검증) 도입 검토

10.4 Cold Start 문제

이력서가 없으면 신규 가입자가 떠나고, 가입자가 없으면 이력서가 쌓이지 않는다.

  • 베타: 본인 + 지인 10~20명에게 직접 요청해 시드 이력서 50개 확보
  • 합격 후기 + 이력서 묶음 콘텐츠로 SEO 자연 유입 노림

10.5 에디터 락인

Tiptap에 깊게 결합하면 후에 Lexical·Plate 등으로 이동 시 비용이 크다.

  • 이력서 데이터를 "섹션 배열 + 도메인 필드 + 부분 Tiptap JSON"으로 저장 (전부 Tiptap JSON에 박지 말 것)
  • 페이지 컴포넌트가 Tiptap API를 직접 호출하지 않게 얇은 래퍼 컴포넌트로 감싸기
