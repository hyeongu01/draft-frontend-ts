Drafted 프론트엔드 개발 워크플로

백엔드(NestJS)는 직접 개발하고, 프론트엔드는 시니어 프론트 에이전트가 만든다. 둘의 계약은 오직 api/swagger.json 하나로 맞추고, 그 파일에서 타입과 TanStack Query 훅을 orval이 자동 생성한다. 이 문서는 그 흐름을 어떻게 굴리는지 설명한다.

핵심 한 줄: 백엔드에서 /docs → api/swagger.json(동기화) → pnpm gen:api(코드 생성) → 화면에서 use… 훅 사용. swagger가 바뀔 때마다 이 사이클을 반복.

이 사이클에서 사람이 직접 타입이나 fetch 코드를 손으로 쓰지 않는다. swagger가 단일 진실 출처이고, 코드는 그로부터 파생된다. 계약에 부족함이 보이면 코드로 우회하지 않고 백엔드에 변경을 요청한다.

왜 커스텀 mutator인가?

이 프로젝트의 인증은 까다롭다 — accessToken은 메모리, refreshToken은 HttpOnly 쿠키, 401이면 자동 refresh 후 재시도, 응답은 { statusCode, timestamp, data } 봉투. 이 로직은 이미 apiJson에 있다. orval 생성 훅이 자체 fetch를 쓰면 이 처리를 우회하므로, 생성 훅 → customFetch → apiJson으로 연결해 모든 호출이 동일한 인증·봉투 처리를 타게 했다. 그래서 생성 훅의 data는 봉투를 벗긴 실제 payload다.

3. 일상 워크플로

A. 백엔드 계약이 새로 생기거나 바뀐 경우

  1. 백엔드(NestJS)를 localhost:4000으로 실행한다.
  2. 채팅에 /sync-swagger 입력 → api/swagger.json이 최신 /docs 응답으로 갱신되고, 이어서 pnpm gen:api로 훅이 재생성된다.
  3. 에이전트가 변경된 엔드포인트·타입의 diff와 영향받는 화면을 요약 보고한다.
  4. 필요하면 화면 코드를 새 훅에 맞춰 정합한다.

B. 화면/기능을 새로 구현하는 경우

  1. 에이전트에게 작업을 맡긴다 (예: "내 이력서 목록 화면 만들어줘").
  2. 에이전트가 api/swagger.json에서 해당 계약을 확인한다.
  3. 계약이 있으면 → 생성 훅으로 구현. 없으면 → 계약을 요청하거나 USE_MOCK으로 임시 진행.
  4. 구현 후 pnpm lint + npx tsc --noEmit로 검증.
순서 주의: swagger를 바꾸면 반드시 pnpm gen:api로 재생성해야 코드에 반영된다. /sync-swagger 스킬은 이 둘을 묶어서 수행한다.

5. 에이전트 사용법

senior-frontend 는 이 프로젝트의 모든 아키텍처 규칙(메모리 토큰, 봉투, CSR/SSR 분리, orval 워크플로)을 알고 있는 시니어 프론트 개발자다. 프론트 작업은 이 에이전트에게 맡긴다.

잘 맡기는 예

  • "공개 이력서 상세 페이지 /resumes/[id] 구현해줘"
  • "좋아요 버튼에 useOptimistic 붙여줘"
  • "swagger 새로 동기화하고 바뀐 부분 반영해줘"
  • "이 화면 생성 훅으로 리팩터해줘"

에이전트가 먼저 되묻는 상황

  • 필요한 엔드포인트/필드가 swagger에 없을 때
  • 응답 구조가 비효율적이거나 봉투가 불일치할 때
  • 기획(drafted_planning.html)을 바꿔야 할 때
  • 개인정보 노출 우려가 있을 때

6. 생성 훅 사용

생성된 훅 이름은 use{Controller}{Operation} 형태다. GET은 useQuery, 그 외(POST/PUT/DELETE)는 useMutation으로 생성된다.

조회 (useQuery)

"use client";
import { useUsersControllerGetMyProfile } from "@/lib/api/generated/users/users";

export default function MePage() {
  // data = 봉투 벗긴 payload. 401 재시도/리다이렉트는 customFetch가 처리.
  const { data: me, isLoading, error } = useUsersControllerGetMyProfile();

  if (isLoading) return <Spinner />;
  return <h1>{me?.nickname}</h1>;
}

변경 (useMutation)

import { useUsersControllerUpdateMyProfile } from "@/lib/api/generated/users/users";
import { useQueryClient } from "@tanstack/react-query";

const qc = useQueryClient();
const { mutate, isPending } = useUsersControllerUpdateMyProfile({
  mutation: {
    onSuccess: () => qc.invalidateQueries(), // 관련 쿼리 무효화
  },
});

mutate({ data: { nickname: "새닉네임" } });
반환 타입이 void로 나온다면? 해당 엔드포인트의 응답 스키마가 swagger에 없는 것이다. 코드로 캐스팅해서 떼우지 말고, 백엔드에 응답 DTO를 문서화해 달라고 요청한다(→ 7장).

임시 래퍼 (swagger에 아직 없는 영역)

resumes·likes·bookmarks처럼 백엔드 미구현 계약은 src/lib/api/*.ts + USE_MOCK 래퍼로 임시 진행하다가, 계약이 swagger에 들어오면 생성 훅으로 교체한다.

7. 계약 변경 흐름 (시니어의 핵심 역할)

프론트에서 계약 부족을 발견하면 코드로 우회하지 않고 다음 형식으로 백엔드(=당신)에게 요청한다:

  1. 왜 필요한지 — 어느 화면/동선에서 무엇이 막히는지
  2. 제안 스키마 — 요청/응답 예시(JSON)
  3. 영향 범위 — 다른 엔드포인트·화면에 미치는 영향

대표적인 요청 트리거:

  • 응답에 content 스키마가 없어 생성 훅이 void → 응답 DTO 명시 요청
  • 봉투({ statusCode, timestamp, data })가 일부 엔드포인트에만 적용 → 전 엔드포인트 통일 요청
  • 목록인데 페이지네이션이 없음 → cursor/page 파라미터 추가 요청
  • 공개 응답에 본명·연락처 등 식별정보 포함 → 제거 요청 (개인정보 정책)
반영 흐름: 백엔드 수정 → /sync-swagger → pnpm gen:api → 화면 정합.

8. 규칙 · 금지

✅ 한다

  • swagger에 있는 건 생성 훅으로
  • swagger 바뀌면 즉시 재생성
  • 계약 부족은 백엔드에 요청
  • 모든 호출 credentials:'include' (customFetch가 처리)
  • OAuth 콜백은 클라이언트 page.tsx

🚫 안 한다

  • generated/ 수동 편집
  • swagger에 없는 엔드포인트 추측 호출
  • 타입·fetch를 손으로 작성(임시 영역 제외)
  • accessToken을 localStorage에 저장
  • @supabase/* 재도입 (마이그레이션으로 제거됨)
